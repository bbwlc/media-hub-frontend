# UC-01.1: Ablauf Registrierung mit Fake-E-Mail (Dev Mode)

## Übersicht

Beschreibt den vollständigen technischen Ablauf der Registrierung, inklusive des simulierten E-Mail-Versands über das Terminal.

---

## Schritt 1 — Frontend-Formular (`app/register/page.tsx`)

- Der Benutzer füllt die Felder **Username**, **Password**, **Confirm Password** und **Email** aus.
- Beim Absenden läuft zuerst die **Client-seitige Validierung**:
  - Pflichtfelder dürfen nicht leer sein.
  - Die E-Mail-Adresse muss dem Format `x@x.x` entsprechen (Regex-Prüfung).
  - Passwort und Passwort-Bestätigung müssen übereinstimmen.
- Schlägt die Validierung fehl, werden Inline-Fehlermeldungen angezeigt — die Anfrage wird **nicht** abgeschickt.
- Bei erfolgreicher Validierung wird die Server Action via `startTransition(() => formAction(formData))` aufgerufen.
- Der Submit-Button ist während der laufenden Anfrage deaktiviert (`pending`-Zustand).

---

## Schritt 2 — Server Action (`app/register/actions.ts`)

- Läuft serverseitig (Next.js Server Action).
- Prüft nochmals, ob Passwort und Bestätigung übereinstimmen.
- Sendet `POST http://localhost:8080/auth/register` an das Spring Boot Backend:

```json
{
  "username": "user1",
  "password": "password123",
  "email": "user1@example.com"
}
```

- Wertet den HTTP-Statuscode aus:

| Status | Bedeutung                  | Reaktion im Frontend                        |
|--------|----------------------------|---------------------------------------------|
| `201`  | Registrierung erfolgreich  | Zeigt "Check your email"-Screen             |
| `400`  | Ungültige Daten            | Fehlermeldung: "Invalid registration data." |
| `409`  | Username bereits vergeben  | Fehlermeldung: "Username is already taken." |
| `500`  | Serverfehler               | Fehlermeldung: "Server error."              |

---

## Schritt 3 — Backend (`AuthController.java`, `POST /auth/register`)

- Validiert Eingaben serverseitig (leere Felder → `400`).
- Prüft, ob der Username bereits existiert (`AccountRepository`) → bei Duplikat `409`.
- Speichert den Account in der H2-Datenbank (Passwort wird gehasht via `PasswordEncoder`).
- **Loggt den registrierten Account und das generierte JWT-Token in die Konsole:**

```
log.info("registered: {}", account);
log.info("tokenData {}", tokenData);
```

- Gibt `201 Created` mit dem JWT-Token im Body zurück.

---

## Schritt 4 — Fake-E-Mail = Terminal-Log

Es gibt **keinen echten E-Mail-Versand**. Der Aktivierungslink wird im **Spring Boot Terminal** ausgegeben (der `log.info`-Eintrag mit `tokenData`).

Das Frontend zeigt nach erfolgreicher Registrierung den Hinweis:

> *(Dev mode: see the activation link in the backend console)*

Um den "Link" zu sehen: im Spring Boot Terminal nach dem `tokenData`-Log-Eintrag schauen.

---

## Schritt 5 — Erfolgs-Screen (`app/register/page.tsx`)

Nach Erhalt von `201` wechselt das Frontend nicht zu `/login` (wie in UC-01 beschrieben), sondern zeigt stattdessen einen **"Check your email"-Screen**:

- Icon: 📬
- Titel: *"Check your email"*
- Text: *"We sent an activation link to your address. Click it to complete your registration."*
- Dev-Hinweis: *(Dev mode: see the activation link in the backend console)*

---

## Zusammenfassung

```
Browser          Next.js (Server)        Spring Boot Backend
  |                    |                         |
  |-- submit form ---->|                         |
  |  (client validation ok)                      |
  |                    |-- POST /auth/register -->|
  |                    |                         | speichert Account
  |                    |                         | loggt Token ins Terminal
  |                    |<-- 201 + JWT -----------|
  |<-- "Check email" --|                         |
```

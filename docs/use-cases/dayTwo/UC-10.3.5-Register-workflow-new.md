# UC-10.3.5: Neuer Registrierungs-Workflow (Double-Opt-In)

## Übersicht

Beschreibt den vollständig überarbeiteten Registrierungs-Workflow nach der Umstellung auf Double-Opt-In (UC-10.3.1–10.3.5). Im Unterschied zum alten Flow wird kein Account sofort angelegt — die Registrierung bleibt solange ausstehend, bis der Benutzer den Bestätigungslink in der E-Mail anklickt.

---

## Geänderte Dateien (Frontend)

| Datei | Änderung |
|---|---|
| `app/register/actions.ts` | `firstname` + `lastname` ergänzt; Erfolgsstatus von `201` auf `202` geändert |
| `app/register/page.tsx` | Felder First name / Last name ergänzt; Validierung erweitert |
| `app/register/confirm/[token]/page.tsx` | Neues: Aktivierungsseite für den Bestätigungslink |

---

## Phase 1 — Registrierungsformular (`/register`)

### Formularfelder

| Feld             | Typ                  | Pflicht | Validierung                          |
|------------------|----------------------|---------|--------------------------------------|
| First name       | `<input text>`       | Ja      | Nicht leer                           |
| Last name        | `<input text>`       | Ja      | Nicht leer                           |
| Username         | `<input text>`       | Ja      | Nicht leer                           |
| Password         | `<input password>`   | Ja      | Nicht leer                           |
| Confirm password | `<input password>`   | Ja      | Muss mit Password übereinstimmen     |
| Email            | `<input email>`      | Ja      | Nicht leer + gültiges E-Mail-Format  |

### Ablauf

1. Benutzer füllt alle Felder aus und klickt Submit.
2. Client-seitige Validierung (`validate()` in `page.tsx`) läuft — bei Fehler: Inline-Fehlermeldungen, kein Request.
3. Server Action (`actions.ts`) sendet `POST /auth/register`:

```json
{
  "firstname": "Max",
  "lastname": "Muster",
  "username": "mmuster",
  "password": "geheim123",
  "email": "max@example.com"
}
```

4. Backend legt **keinen Account** an, sondern erstellt eine `PendingRegistration` (im Arbeitsspeicher) und gibt `202 Accepted` zurück.
5. Frontend zeigt den "Check your email"-Screen.

### Backend-Antworten

| Status | Bedeutung | Reaktion Frontend |
|---|---|---|
| `202 Accepted` | Registrierung ausstehend, E-Mail unterwegs | Erfolgs-Screen anzeigen |
| `400 Bad Request` | Ungültige Eingabedaten | Fehlermeldung: *"Invalid registration data."* |
| `409 Conflict` | Username bereits vergeben | Fehlermeldung: *"Username is already taken."* |
| `500` | Serverfehler | Fehlermeldung: *"Server error. Please try again later."* |

---

## Phase 2 — Fake-E-Mail (Dev Mode)

Der `MailService` im Backend sendet **keine echte E-Mail**, sondern gibt den Aktivierungslink auf der Konsole aus:

```
Registrierungslink fuer max@example.com: http://localhost:8080/auth/register/confirm/<uuid>
```

> Für eine vollständige Frontend-Integration sollte dieser Link auf die Frontend-URL zeigen:
> `http://localhost:3000/register/confirm/<uuid>`
> (erfordert Anpassung im Backend-`MailService`)

---

## Phase 3 — Aktivierungsseite (`/register/confirm/[token]`)

Der Benutzer klickt den Link aus der E-Mail (resp. dem Terminal). Die Seite ist ein **Server Component** (`app/register/confirm/[token]/page.tsx`), das beim Laden sofort den Backend-Endpoint aufruft:

```
GET http://localhost:8080/auth/register/confirm/{token}
```

### Mögliche Zustände

| Backend-Status | Bedeutung | Anzeige |
|---|---|---|
| `201 Created` | Account erfolgreich angelegt | ✅ "Account activated!" + Button "Sign in" → `/login` |
| `404 Not Found` | Token unbekannt oder bereits verwendet | ❌ "Invalid link" + Link "Register again" → `/register` |
| `410 Gone` | Token abgelaufen (älter als 24h) | ⏰ "Link expired" + Link "Register again" → `/register` |
| Netzwerkfehler | Backend nicht erreichbar | ⚠️ "Something went wrong." |

## Gesamtablauf

```
Browser (Frontend)          Next.js (Server Action)      Spring Boot Backend
        |                           |                             |
        |-- Formular absenden ----->|                             |
        |   (Client-Validierung ok) |                             |
        |                           |-- POST /auth/register ----->|
        |                           |   { firstname, lastname,    |
        |                           |     username, password,     |
        |                           |     email }                 |
        |                           |                             | PendingRegistration anlegen
        |                           |                             | Token generieren (UUID)
        |                           |                             | MailService: Link → Konsole
        |                           |<-- 202 Accepted ------------|
        |<-- "Check your email" ----|                             |
        |                           |                             |
        |  (Link aus Terminal kopieren und aufrufen)              |
        |                           |                             |
        |-- GET /register/confirm/{token} -->|                    |
        |                           |-- GET /auth/register/confirm/{token} -->|
        |                           |                             | PendingRegistration suchen
        |                           |                             | Gültigkeit prüfen (24h)
        |                           |                             | Account in DB anlegen
        |                           |                             | PendingRegistration entfernen
        |                           |<-- 201 Created + JWT -------|
        |<-- Aktivierungsscreen ----|                             |
        |   ✅ "Account activated"  |                             |
        |   [Sign in] → /login      |                             |
```

---

## Unterschied zum alten Workflow

| Aspekt | Alt | Neu |
|---|---|---|
| Backend-Antwort auf Register | `201 Created` + JWT | `202 Accepted` (kein JWT) |
| Account in DB | Sofort beim `POST /auth/register` | Erst nach Klick auf Bestätigungslink |
| Felder im Formular | Username, Password, Confirm, Email | + First name, Last name |
| Bestätigung | Kein Confirm-Schritt | `GET /register/confirm/{token}` |
| Token-Ablauf | Nicht vorhanden | 24h, danach `410 Gone` |
